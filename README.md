# [騎行數據整合與保養監控系統]

## Proposal Report

### 動機與目標
<!-- 說明為什麼想做這個專題 -->
雖然單車的耗材更換與否大多是根據外觀來決定(比如直接測量鍊條拉伸程度、外胎磨損程度)，但不是每個人上路前都會仔細檢查車輛，所以我想透過實際的騎乘數據來了解一項耗材的使用距離，提供直觀的對比(比如A外胎6000公里就不行了，B可以撐10000公里)。

### 競品比較
<!-- 比較目前已經存在可取得的類似工具或應用 -->
有許多騎行數據紀錄工具如strava、velodash...等，但兩者都缺乏針對耗材的里程累積，也缺乏針對旅程的文字、圖片記錄區塊。
### 預期功能
<!-- 列出預計實作的功能 -->
實作出針對耗材的里程累積(鍊條、鍊條油、外胎、把帶、煞車皮、線材、保養)，以及視覺化從strava抓來的數據，提供類似日記的紀錄方式。
### 使用技術
<!-- 使用的語言、框架、工具等 -->
本專案採用典型的 全端 Web 架構，結合第三方 API 服務實現數據驅動的應用：

後端框架 (Backend)：使用 Python Flask。其輕量化特性適合快速開發 Prototype，並能有效處理 API 請求與數據運算。

第三方串接 (API Integration)：Strava V3 API。透過 OAuth 2.0 協議進行身分驗證，確保在符合資安規範的前提下獲取使用者的騎行紀錄。

數據處理 (Data Processing)：利用 Python 內建的 datetime 與 requests 模組，將 API 獲取的原始 Unix 時間戳與公尺單位，轉換為具備可讀性的「騎行日記」格式。

前端介面 (Frontend)：

Jinja2：動態模板引擎，將後端處理完的活動列表渲染至前端日記區塊。

數據視覺化 (Data Visualization)：Chart.js。將繁雜的騎行條目轉化為直觀的里程趨勢圖，模擬專業運動平台的數據呈現質感。
### Prototype 預計可驗證內容
自動化數據同步功能 (API Sync)：

實作內容：串接 Strava OAuth 2.0 API。

驗證點：使用者登入後，系統能自動抓取真實的騎行活動（Activity），而非使用寫死的假數據。

動態里程累計與保養演算法 (Maintenance Logic)：

實作內容：開發「特定時間點後」的里程加總邏輯。

驗證點：當使用者點擊「標記已上油」後，系統能將累積里程歸零，並從該時刻起重新計算，證明保養週期追蹤邏輯正確。

自定義耗材門檻控制 (Custom Threshold)：

實作內容：前端表單輸入與後端變數同步。

驗證點：使用者可以自行設定鏈條（如 600km）或外胎（如 5000km）的更換標準，系統進度條會隨之調整比例。

資訊轉譯與日記呈現 (Data Transformation)：

實作內容：將原始 API 數據轉換為「日記卡片」。

視覺化趨勢分析 (Trend Visualization)：

實作內容：整合 Chart.js 繪製折線圖。

驗證點：圖表能根據近期的騎行頻率自動畫出里程起伏，證明能提供使用者宏觀的運動量分析。

## Prototype Report

### 目前進度
<!-- 完成了什麼 -->
多耗材分流監控系統：成功擴充監控維度，除了原有的鍊條保養外，新增了「外胎壽命」追蹤邏輯。系統現在能針對不同零件設定獨立的里程閾值（如鍊條 $600\text{km}$、外胎 $5000\text{km}$），並分別計算其累積里程。資料持久化 (Persistence) 實作：導入了 JSON 檔案儲存機制（user_data.json）。系統現在具備記憶功能，能永久儲存使用者的保養日期與自定義門檻，解決了先前程式重啟後數據歸零的問題。模組化重構：優化了後端邏輯，將里程計算封裝成通用函式 get_accumulated_km()，大幅提升了未來新增其他耗材（如煞車皮、飛輪）的開發效率。動態管理介面：實作了獨立的 POST 路由處理器，讓使用者能針對特定耗材進行單獨的「保養歸零」操作。
### 遇到的困難
<!-- 遇到什麼問題、如何解決或打算如何解決 -->
程式碼維護與語法衝突：

困難：在進行頻繁的功能更新與代碼段落合併時，發生了 SyntaxError（如重複導入模組或語法結構不完整），導致系統無法正常啟動。

解決方法：重新整理 dsap.py 的檔案結構，將模組導入、功能定義與啟動邏輯（if __name__ == '__main__':）嚴格分離，確保程式碼的穩定性。

資料讀寫異常處理：

困難：初次執行時若 user_data.json 檔案不存在，會導致讀取錯誤。

解決方法：在 load_data() 函式中加入 os.path.exists 檢查與 try-except 錯誤捕捉機制，確保檔案缺失或格式錯誤時能自動初始化預設數值。

## Final Report

### 專案說明
<!-- 完整描述你的專案做了什麼 -->
一、開發動機與背景
對於公路車愛好者而言，維持車輛的最佳狀態是確保騎行安全與性能表現的關鍵。鏈條、外胎、煞車皮等核心耗材的壽命，不僅取決於累積騎乘里程，也深受安裝放置時間（如橡膠老化、潤滑油品變質）的影響。

然而，現有的運動平台（如 Strava、Garmin）多半僅提供單一的累積里程統計，缺乏時間維度的追蹤；市面上的車輛管理工具也普遍缺乏自動化數據同步，導致車友必須手動登錄里程，極易產生「耗材焦慮」或因疏忽保裝而導致機械故障。

本專案開發之 RideMonitor Pro 騎行數據整合與保養監控系統，旨在建立一個自動化、雙軌制的單車資產與耗材管理平台，完美解決上述痛點。

二、系統核心定位與功能
本系統以「數據整合」與「智慧預警」為核心，主要功能包含：

自動化數據異質整合：
對接國際主流單車運動平台 Strava，利用 OAuth 2.0 認證協議非同步抓取使用者的真實騎乘里程與爬升數據，免去手動輸入的繁瑣流程。

「里程與時間」雙軌制預警演算法：
針對每項耗材設定獨立的里程與時間壽命閥值，後端採用 OR 邏輯進行狀態判定。系統會同時計算「里程消耗百分比」與「時間消耗百分比」，並取兩者之中的最大值作為當前狀態。當百分比達到或超過 85% 時，系統將即時觸發 Warning 警報，提示使用者進行檢修或更換。

動態數據視覺化儀表板：
整合 Chart.js，將後端聚合的歷史騎乘數據轉化為直觀的逐月里程統計圖，幫助使用者掌握自身騎乘週期。

個人化騎行日記與相簿管理：
提供具備隱私管理（動態隱藏與復原）功能的 Feed 流日記，並根據 Strava 匯入的單次騎乘強度，自動進行智慧標籤黏貼（如高強度或休閒騎），打造專屬的單車日誌。

三、技術架構
本專案基於輕量化與高效開發原則，採用以下技術棧構建：

後端框架：
Python Flask 微框架，負責路由控制、API 請求處理與業務邏輯運算。

資料持久化：
採用本機檔案系統管理。結構化數據（耗材設定、日記文字、路徑索引）採用 JSON 格式（user_data.json）進行儲存；二進位圖片檔案則上傳至伺服器相對路徑之靜態資料夾（static/uploads/）儲存。

前端介面：
Bootstrap 5 響應式網頁設計、Vanilla JS、Chart.js 視覺化圖表庫。
### 使用方式
<!-- 如何編譯、執行、使用你的程式 -->
一、環境準備與建置
本專案基於 Python 3 環境開發，執行前請確保系統已安裝 Python，並透過終端機（Terminal 或命令提示字元）執行以下指令安裝所需相依套件：
指令：pip install flask requests werkzeug

二、專案執行

將主程式命名為 dsap.py。

在終端機切換至該程式碼所在的專案資料夾目錄，執行以下指令啟動本機伺服器：
指令：python dsap.py

啟動成功後，終端機會顯示服務已成功運行，請保留終端機視窗請勿關閉。

三、系統操作流程說明

平台授權與數據同步：
開啟瀏覽器並於網址列輸入http://localhost:5000。進入首頁後，點擊「透過 Strava 數據同步登入」按鈕。系統將引導至 Strava 官方的 OAuth 2.0 授權網頁，使用者同意授權後，系統會自動在後端非同步抓取該帳號的真實騎乘歷史數據。

儀表板總覽（Dashboard）：
完成登入後會自動跳轉至主儀表板。
上方卡片會動態計算並呈現平台總累積騎乘里程與總爬升高度。
左側面板可點擊「編輯愛車資料」，供使用者自定義單車名稱並上傳多張愛車照片（支援輪播功能）；下方設有警報專區，當有任何耗材的綜合損耗率達到 85% 以上，將會動態跳出紅色 Warning 提示框。
右側區塊則透過 Chart.js 視覺化圖表，直觀展示使用者過去 12 個月的逐月累積騎乘里程。

耗材保養系統（Maintenance）：
透過導覽列切換至「耗材保養系統」，此處為本專案的核心管理介面。
左側列表會顯示所有追蹤中的零件項目，並渲染出獨立的「里程」與「時間」雙軌進度條。右上角會以「Warning：百分比」顯示目前的最高危險指標。
每個項目皆支援「重設基準日」功能。當使用者更換零件後，可點擊「➔ 標記保養更換」將基準日更新為今日並歸零計時；若不慎誤觸，亦可點擊「↩ 復原重置」還原至上一次的保養紀錄。
右側面板提供「新增自定義監控項目」功能，可自由勾選是否啟用時間週期判定；下方則支援「批量修改現有閥值」，可一次性調整所有零件的公里數與天數上限。

數據騎行日記（Diary Feed）：
透過導覽列進入「騎行日記 Feed」，系統會自動列出所有的單次騎乘活動。
後端演算法會根據單次行程的里程與爬升高度，自動為該次騎乘貼上「高強度」、「中強度」或「休閒騎」的分類標籤。
使用者可點擊「✏ 編輯筆記」進入編輯頁面，撰寫文字心得並上傳當天的本機風景照片。
介面右上角提供「全部 Feed」與「隱藏項目」的切換按鈕。使用者可對特定的行程點擊「標記隱藏」，該行程便會從公開的 Feed 流中移出，確保個人隱私，並可隨時到隱藏項目面板中點擊「取消隱藏」進行復原。
